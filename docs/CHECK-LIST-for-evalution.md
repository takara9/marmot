## 外部評価前のチェックリスト

優先度の軸は固まりました。外部評価を止める要因は、機能数よりも「クラッシュする」「導入で詰む」「ホスト側を壊す」の3つです。

知人に触ってもらう前提なら、優先度は「機能追加」ではなく「試してすぐ壊れないこと」に寄せるべきです。順番に潰すなら次です。

1. パニックでプロセスが落ちる経路の除去  
db-image.go と db-servers.go は、DB更新失敗時に panic します。外部評価でこれが出ると、個別機能の評価以前に「不安定」という印象で終わります。まずは panic をやめて、FAILED 状態と明示的なメッセージに落とすべきです。

2. ネットワーク作成とIP払い出しの信頼性修正  
db-ipam.go は静的解析でも条件不整合と到達不能コードが出ています。ここは server create や network create の根幹なので、少しでも不安定だと最初の体験が崩れます。単一ノード評価でも最優先です。

3. 起動前チェックの整理  
api-core.go は初期化失敗でそのまま終了します。既知メモを見る限り、etcd 未起動、OVS 未起動、DNS の 53 番ポート競合は「入れてすぐ動かない」系の代表です。パッケージ側である程度対策は入っていますが、評価前に最低でも「必須サービス確認」「競合ポート確認」「不足時の分かるエラー」は揃えたほうがいいです。

4. 内部 DNS まわりの安全性確認  
既知の論点として、DNS bind に失敗した状態で resolver を切り替えると、ホスト側の名前解決まで壊すリスクがあります。外部評価でホスト環境を壊すのは避けるべきなので、この系統は「失敗してもホストを壊さない」ことを確認してから出したほうがいいです。関連実装は dns サーバー と resolver 更新 です。

5. CLI の未完成部分を最低限そろえる  
main.go が常時有効で、server_update.go、image_update.go、volume_rename.go は text 出力が未実装です。致命傷ではないですが、試用者に渡すなら「出力が中途半端」「デバッグ臭い」は早めに消したほうが評価が安定します。

6. マルチノード、VXLAN、LB、Gateway は評価対象から外す  
この領域は実装は進んでいますが、既知メモ上も統合テストや運用検証がまだ主題です。今の段階で知人評価に入れると、基本機能ではなく周辺機能の不安定さで全体評価が下がります。まずは単一ノード、Ubuntu 24.04、server/network/image の基本作成・一覧・削除に絞るのが妥当です。

逆に、すでに優先度を下げてよさそうなものもあります。etcd と OVS の postinst 起動、volume group の設定化、default network の DNS 退行は、既知メモ上は対策済みです。ここは「未修正 issue」として追うより、「配布予定の deb に本当に入っているか」を確認する工程に回すほうがいいです。

短く言うと、知人に渡す前の最低ラインは 1 から 4 です。ここを越えていれば試用は可能です。5 は印象改善、6 は評価範囲の切り分けです。

1. この優先度をそのまま GitHub Issue 化できる粒度に分解する
2. 単一ノード評価用のチェックリストを作る
3. 今のコードから 1 と 2 の修正候補を具体化する

のどれかまで進めます。
