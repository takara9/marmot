**公開前チェックリスト**

以下をすべて確認できたら、OSS として公開してよい可能性が高いです。

**1. 権利・ライセンス**
- □ 生成コードに、既存 OSS のコードをそのまま含んでいない
- □ 既存コードに似すぎている箇所があれば、書き直した
- □ 依存ライブラリのライセンスを確認した
- □ GPL など、公開方針と相性が悪いライセンスが混ざっていない
- □ 必要な LICENSE、NOTICE、著作権表示を入れた
- □ 会社や個人の規約上、公開して問題ない

**2. 秘密情報・個人情報**
- □ API キー、トークン、パスワードが残っていない
- □ 内部 URL、社内ホスト名、IP、リポジトリ情報が残っていない
- □ ログやサンプルデータに個人情報が含まれていない
- □ `.env` や設定ファイルの扱いが公開向けになっている
- □ 履歴にも機密情報が残っていない

**3. 品質**
- □ 主要機能のテストが通る
- □ エラー処理が不足していない
- □ 境界値、空値、異常系を確認した
- □ 生成コードのまま理解しづらい箇所を整理した
- □ 主要な lint や format の警告が解消されている
- □ README だけで動かし方が分かる

**4. セキュリティ**
- □ 入力値の検証がある
- □ 認証・認可の前提が明確
- □ 危険なコマンド実行やシェル展開がない
- □ ファイル操作やパス操作に脆弱性がない
- □ 依存関係の既知脆弱性を確認した
- □ デフォルト設定が安全寄りになっている

**5. OSS としての使いやすさ**
- □ 何をするプロジェクトか README に書いた
- □ インストール手順が再現できる
- □ サンプル設定や利用例がある
- □ Issue テンプレートや Contributing 方針があると望ましい
- □ サポート範囲と非対応範囲を明記した
- □ 破壊的変更の可能性があれば注意書きを入れた

**6. Copilot 由来コードの確認**
- □ 生成されたままの大きな塊を、そのまま採用していない
- □ 人間のレビューを通して意味を確認した
- □ 実装意図を自分で説明できる
- □ 似た実装が外部にないか軽く調べた
- □ 最終責任は自分たちが負える状態になっている

**公開直前の最終確認**
- □ `git status` が意図した変更だけになっている
- □ ビルドとテストが通る
- □ README とライセンスを最終確認した
- □ タグやリリースノートを付ける準備ができている


---
実務では、1つの方法だけで断定せず、複数の観点で重ねて確認するのが安全です。

- まず、生成コードの特徴が強い箇所を洗い出します。長い関数、独特なコメント、変数名、エラーメッセージ、並び順が不自然に整っている部分は要注意です。
- 次に、そのままの文字列で広く検索します。エラーメッセージ、コメント、関数名、特殊な定数などを手がかりに、Web 検索やリポジトリ内検索で一致先がないか見ます。
- 似ている実装が見つかったら、構造レベルで比較します。単なる一般的なアルゴリズムなのか、行順や分岐、コメントまで近いのかを見ます。後者なら書き直し候補です。
- OSS のライセンススキャンをかけます。依存関係だけでなく、サンプルコードやベンダーコードを取り込んでいないかも確認します。
- コピペ臭の強い部分は、意図を変えずに自分の設計で書き直します。単語の置換だけでは不十分で、制御構造や分割の仕方も変えるのが安全です。
- 重要な箇所は、同僚や第三者に「この実装はどこ由来か」を説明できるかで最終確認します。説明できないなら、公開前に再検討したほうがよいです。

自動化するなら、次の組み合わせが実用的です。  
- 開発者ツールでの広域文字列検索  
- 依存関係のライセンススキャン  
- 類似コード検索サービスの利用  
- 重要ファイルの手動レビュー

判定基準としては、次のどれかに当てはまると危険度が高いです。  
- 元コードとコメントや命名までほぼ一致している  
- 分岐や例外処理の順番が特徴的に一致している  
- 独特な定数や文言がそのまま残っている  
- 説明を求められても、出自を明確に説明できない

必要なら次に、あなたのリポジトリ向けに「公開前に実際に使える確認コマンド一覧」まで落とし込めます。

## 公開前に実際に使える確認コマンド一覧

このリポジトリでは、まず次の順で確認すると効率的です。

```bash
# 1. 生成コードに特徴的な文言や既存実装の断片が残っていないか探す
rg -n --hidden --glob '!dist/**' --glob '!vendor/**' 'TODO|FIXME|Copyright|Licensed under|All rights reserved' .

# 2. 似た実装や既存 OSS の痕跡を拾うために、怪しい関数名やエラーメッセージで再検索する
rg -n --hidden --glob '!dist/**' --glob '!vendor/**' '調べたい文字列' .

# 3. 主要テストを走らせる
go test ./...

# 4. このリポジトリの標準チェックをまとめて通す
make test

# 5. パッケージ作成まで問題ないか確認する必要があるとき
CGO_ENABLED=1 make package
```

補足として、DB 周辺だけを見たい場合は次も使えます。

```bash
cd pkg/db
go test -v .
```

判定の目安は、検索で元コード特有の文言や構造が出てこないこと、`go test ./...` と `make test` が通ること、そして怪しい箇所を第三者に説明できることです。

OSSライセンススキャンの具体的な作業は、大きく分けて以下のような行為になります。

## 1. 依存関係の洗い出し(インベントリ作成)

まず使用しているOSSコンポーネントを全て特定します。

- 直接依存だけでなく**推移的依存(依存先の依存先)**も含める
- `package.json`/`go.mod`/`pom.xml`/`requirements.txt`などのマニフェストファイルを解析
- コンテナイメージの場合はOSレイヤーのパッケージ(apt/yumでインストールされたもの)も対象
- バイナリに静的リンクされたライブラリも(可能な範囲で)検出

この結果を**SBOM(Software Bill of Materials)**として出力するのが現在の標準的なやり方です。形式はSPDXやCycloneDXが主流です。

## 2. ライセンス種別の特定

検出した各コンポーネントについて、どのライセンスが適用されているかを判定します。

- LICENSEファイル、ヘッダーコメント、パッケージメタデータから抽出
- 表記が曖昧な場合(「Apache-2.0 OR MIT」のようなデュアルライセンスなど)の解釈
- ライセンスが不明・未記載のパッケージの扱い(要手動確認)

## 3. ポリシーとの照合(コンプライアンスチェック)

組織で定めた「使用可否ルール」と照合します。

- **許可リスト方式**: MIT, Apache-2.0, BSD系などビジネス利用しやすいものはOK
- **禁止/要注意リスト**: GPL系(特にAGPL)はコピーレフトの影響範囲が広いため、配布形態によっては自社コードの開示義務が発生するリスクがあり、要審査とすることが多い
- ライセンスの互換性問題(あるライセンスと別のライセンスを同時に満たせない組み合わせ)の検出

## 4. 脆弱性スキャンとの連携(運用上よく一緒にやる)

ライセンススキャンツール(Trivy, Snyk, FOSSAなど)は脆弱性スキャンも同時に行えることが多く、SES/Kubernetes環境では同じパイプラインで両方回すのが一般的です。

## 5. 結果のレポーティングと例外管理

- 検出された違反/要注意ライセンスのレポート生成
- 法務・コンプライアンス部門への報告フロー
- 「このパッケージは例外的に許可する」といったWhitelist/Exception管理
- 監査証跡として履歴を残す

## 6. 表示義務への対応(配布する場合)

製品として配布する場合、ライセンス条文によっては以下が必要になります。

- 著作権表示・ライセンス全文の同梱(NOTICE/THIRD-PARTY-LICENSESファイルなど)
- ソースコード開示義務がある場合の対応(コピーレフト系)

## 7. CI/CDへの組み込み(継続的スキャン)

一度きりではなく、コード変更・依存追加のたびに自動検査する仕組みが重要です。

- PRごとに新規依存のライセンスをチェック
- 違反があればビルドを失敗させる(Fail the build)
- 定期スキャン(依存の状態は変わらなくても、ポリシー側が変わることもあるため)
